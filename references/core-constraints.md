# 核心约束

## 用户交互协议（双模式）

所有需要用户确认的环节，必须根据当前运行环境选择交互方式。**严禁两种模式都不使用就直接跳过。**

### 模式判断

首先检查当前环境是否支持 `AskUserQuestion` 工具：
- 如果工具列表中有 `AskUserQuestion` → 使用**模式 A：结构化提问**
- 如果工具列表中没有 `AskUserQuestion` → 使用**模式 B：纯文本多轮对话**

### 模式 A：结构化提问（支持 AskUserQuestion 时）

使用 `AskUserQuestion` 工具向用户提问，提供清晰的选项让用户选择。

格式如下：
```
问题：清晰描述当前议题和需要用户决策的内容
选项：选项1（描述）/ 选项2（描述）
```

### 模式 B：纯文本多轮对话（不支持 AskUserQuestion 时）

在文本回复中清晰列出当前议题和选项，**在回复末尾明确等待用户回复**，然后停止输出，等待用户的下一轮消息。

格式如下：
```
## 需要你确认：[议题名称]

当前情况：[简要描述]

选项：
1. [选项1] — [说明]
2. [选项2] — [说明]

请回复你的选择（如 "1" 或 "选项1"），我会继续下一步。
```

**关键规则**：
- 模式 B 下，每次只抛出一个议题，不要一次性列出所有待确认事项。
- 在用户回复之前，**必须停止输出**，不得继续执行后续步骤。
- 收到用户回复后，确认用户的选择，再进入下一个议题。

### 两种模式通用规则

- 每次只处理一个核心议题，避免一次性抛出过多提问。
- 典型节奏：代码审查 → 等待确认 → 版本号决策 → 等待确认。
- 在未获得明确指令前，绝不擅自修改代码，也绝不进入下一议题。

---

## 只读诊断

除非用户明确授权（如说"同意"、"Apply"、"改吧"），否则绝对不能修改代码。

审查阶段的所有操作仅限于：
- 读取文件
- 分析代码
- 列出问题
- 提出建议

## 安全阻断

若发现严重逻辑错误或安全隐患，必须：
1. 明确指出问题位置和风险。
2. 标注为 🔴 严重级别。
3. 建议阻断提交，直到问题修复。

## 澄清义务（重要）

在执行本指南的任何阶段，如果遇到以下情况，必须立即向用户提问寻求澄清：

- 指令模糊，无法确定审查范围
- 缺失文件，无法验证完整性
- 逻辑矛盾，需求之间互相冲突
- 任何不确定的情况

**宁可不执行，不可乱执行。**
