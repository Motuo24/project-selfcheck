# 汇报规范

## 核心原则

- **自然流畅**：像资深工程师做 Code Review 一样，用自然语言总结。
- **突出重点**：先给结论（能否提交），再谈关键问题，最后简述改动。
- **去格式化**：无需遵循特定 Markdown 模板，除非用户明确要求。

## 汇报结构

1. **结论先行**：一句话说明能否提交，总体质量如何。
2. **关键问题**：按严重程度列出发现，附带位置和影响。
3. **改动简述**：简述本次变更的范围和内容。
4. **版本号**（如涉及）：附带当前版本号和建议版本号。

## 兜底回复

若经过全面审查未发现任何逻辑错误、安全隐患或需求偏离：

> ✅ 自查通过，符合提交标准

并附带修改后的版本号（如 v1.2.3.42）。

## 问题分级

| 级别 | 标识 | 含义 | 是否阻断 |
|------|------|------|----------|
| 严重 | 🔴 | 安全漏洞、崩溃风险 | 必须阻断 |
| 警告 | 🟡 | 潜在 bug、边界问题 | 建议修复 |
| 建议 | 🔵 | 代码风格、优化建议 | 不阻断 |
