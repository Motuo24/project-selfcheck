# 版本号决策规则

## 用户交互方式

**必须遵守 `references/core-constraints.md` 中的「用户交互协议（双模式）」**：
- 支持 `AskUserQuestion` 工具 → 使用结构化提问
- 不支持 → 使用纯文本多轮对话，每次只抛一个议题，等用户回复后再继续

## 版本号格式

`vX.Y.Z.B`

| 位置 | 名称 | 含义 | 变更时机 |
|------|------|------|----------|
| X | Major | 主版本 | 用户特殊说明时更改，通常不变 |
| Y | Minor | 功能更新 | 新增功能，向后兼容 |
| Z | Patch | 修复/优化 | 修 Bug 或优化，无新功能 |
| B | Build | 构建号 | **无论何种变更，每次提交都必须递增** |

## 决策流程（必须遵守）

**整个流程分四步，每步之间必须停下来等用户确认。禁止一次性完成所有步骤。**

### 步骤 1：判断版本号

```
代码是否有变更？
├── 无变更 → 版本号不变，不需要走版本决策流程
└── 有变更 →
    ├── 判断 X / Y / Z 是否需要变更
    │   ├── 新增功能 → Y+1，Z 归零
    │   ├── 修 Bug / 优化 → Z+1
    │   └── 无功能变更 → X/Y/Z 不变
    └── B 号必须递增
        ├── 有 Git 仓库 → 执行 git rev-list --count HEAD 获取
        └── 无 Git 仓库 → 当前 B+1
```

### 步骤 2：确认版本号（必须向用户确认）

判断完成后，**必须立即向用户确认**（按双模式协议执行）：

- 议题："根据本次变更（XXX），建议版本号从 vA.B.C.D 变更为 vX.Y.Z.B，是否同意？"
- 选项：
  - 同意，按此版本号
  - 不同意，我手动指定（让用户输入）

**用户未确认前，不得执行步骤 3。**

### 步骤 3：同步文件（必须向用户确认）

**仅在用户确认版本号后，才进行此步骤。**

检查以下文件是否需要同步版本号（根据项目类型判断）：
- `package.json`（Node.js 项目）
- `AndroidManifest.xml` 或 `build.gradle`（Android 项目）
- `Info.plist` 或 `project.pbxproj`（iOS 项目）
- 其他包含版本号的项目文件

若存在需要同步的文件，**必须立即向用户确认**（按双模式协议执行）：
- 议题："检测到以下文件需要同步版本号：[列出文件路径]，是否同步修改？"
- 选项：
  - 同步所有文件
  - 跳过，暂不修改

**用户未确认前，不得修改任何文件。**

### 步骤 4：更新日志（必须向用户确认）

**仅在用户确认文件同步后（或跳过同步），才进行此步骤。**

检查是否存在：
- 更新公告（首次启动弹窗）
- 更新历史（关于页更新日志 / CHANGELOG.md）

若存在，**必须立即向用户确认**（按双模式协议执行）：
- 议题："检测到项目有更新日志/公告，建议更新内容如下：[内容]，是否需要更新？"
- 选项：
  - 更新
  - 跳过，暂不更新

**用户未确认前，不得修改任何日志文件。**

## 关键规则

- **B 号是唯一每次必变的字段。**即使只是修一个拼写错误，Z 不动，B 也必须 +1。
- **每个步骤之间必须停下来等用户确认。**严禁一次性输出版本号、文件同步、日志更新三个问题。
- **所有修改必须在用户明确同意后才执行。**不得擅自改文件。

## 示例

| 场景 | 变更前 | 变更后 | 说明 |
|------|--------|--------|------|
| 修 Bug | v1.2.3.41 | v1.2.3.42 | 仅 Z+1，B 用 git count |
| 新增功能 | v1.2.3.41 | v1.3.0.42 | Y+1，Z 归零，B 递增 |
| 仅改配置 | v1.2.3.41 | v1.2.3.42 | X/Y/Z 不变，B 仍递增 |
| 无代码变更 | v1.2.3.41 | v1.2.3.41 | 不变 |
