# 安全审计

## 触发条件

- 用户明确要求"安全检查""安全审计"
- 代码涉及认证、授权、数据加密、用户输入
- 处理敏感数据（个人信息、支付、凭证）

## 用户交互方式

**必须遵守 `references/core-constraints.md` 中的「用户交互协议（双模式）」**：
- 支持 `AskUserQuestion` 工具 → 使用结构化提问
- 不支持 → 使用纯文本多轮对话，每次只抛一个议题

## 审计维度

### 1. 密钥与凭证

- 是否硬编码 API Key、Token、密码、私钥
- 是否将凭证提交到代码仓库
- 环境变量 / 密钥管理服务是否规范使用

### 2. 注入攻击

- SQL 查询是否使用参数化
- 用户输入是否直接拼接到命令行
- HTML/JS 中是否直接插入用户输入（XSS）
- 路径拼接是否存在目录穿越风险

### 3. 权限与认证

- 接口是否有权限校验
- Token 过期/刷新机制是否完善
- 是否有水平/垂直越权风险
- 敏感操作是否有二次确认

### 4. 数据安全

- 敏感数据是否加密传输（HTTPS）
- 敏感数据是否加密存储
- 日志中是否包含敏感信息
- 错误信息是否暴露内部细节

## 审计结果分级

| 级别 | 标识 | 含义 |
|------|------|------|
| 严重 | 🔴 | 可直接导致数据泄露或系统被入侵 |
| 高危 | 🟠 | 存在明确攻击路径 |
| 中危 | 🟡 | 存在潜在风险，需要特定条件 |
| 低危 | 🔵 | 最佳实践建议 |

## 工作流

1. 扫描变更文件中的安全敏感代码。
2. 按审计维度逐项检查。
3. 发现问题立即标注级别并列出攻击路径。
4. 严重/高危问题必须阻断提交。
5. 按双模式协议向用户确认处理方案。
