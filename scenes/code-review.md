# 代码审查专项

## 触发条件

- 用户明确要求"代码审查""review 一下代码"
- 用户说"帮我看看这段代码有没有问题"
- 上下文中有明确的代码变更需要审查

## 用户交互方式

**必须遵守 `references/core-constraints.md` 中的「用户交互协议（双模式）」**：
- 支持 `AskUserQuestion` 工具 → 使用结构化提问
- 不支持 → 使用纯文本多轮对话，每次只抛一个议题

## 审查维度（按优先级排序）

### 1. 安全

- 是否硬编码密钥、Token、密码、API Key
- 是否存在 SQL 注入、XSS、命令注入风险
- 权限校验是否完整，是否存在越权漏洞
- 敏感数据是否明文传输或存储

### 2. 逻辑

- 空值/undefined 处理是否完善
- 边界条件是否覆盖（空数组、零值、极值）
- 异步操作是否有正确的错误处理（try/catch、.catch）
- 异常路径是否会导致状态不一致
- 循环/递归是否有终止条件

### 3. 回归

- 改动是否影响现有功能
- 公共 API/接口签名是否向后兼容
- 是否有未更新的调用方

### 4. 需求符合度

- 是否实现了用户要求的全部功能
- 是否有遗漏的边缘情况
- 输入输出是否符合预期

详细检查清单见 `references/review-checklist.md`。

## 工作流

1. 读取变更文件，逐个审查。
2. 发现问题时，按严重程度分类：🔴 阻断（安全/崩溃）、🟡 警告（潜在 bug）、🔵 建议（优化）。
3. 按双模式协议向用户确认，逐个请示，等待指令。
4. 汇总审查结果，按 `references/reporting-standards.md` 汇报。
