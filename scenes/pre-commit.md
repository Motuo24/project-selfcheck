# 提交前全面自查

## 触发条件

- 用户执行 `/project-selfcheck` 指令
- 用户说"提交前检查一下""帮我自查""自纠一下"
- 用户准备 git commit 前要求审查

## 用户交互方式

**必须遵守 `references/core-constraints.md` 中的「用户交互协议（双模式）」**：
- 支持 `AskUserQuestion` 工具 → 使用结构化提问
- 不支持 → 使用纯文本多轮对话，每次只抛一个议题，等用户回复后再继续

以下各阶段中"向用户确认"的环节，均按此协议执行。

## 工作流（分步执行，每步必须等用户确认后才能继续）

整体分为四个阶段，每个阶段之间必须停下来等待用户确认。**严禁跨阶段一次性输出所有结果。**

```
阶段一：代码审查
  → 审查完毕 → 向用户确认（是否同意修复？）
  → 用户确认 → 进入阶段二

阶段二：版本号决策
  → 计算出建议版本号 → 向用户确认（这个版本号可以吗？）
  → 用户确认 → 进入阶段三

阶段三：前端同步
  → 检查是否需要同步文件 → 向用户确认（需要同步这些文件吗？）
  → 用户确认 → 进入阶段四

阶段四：更新日志
  → 检查是否需要更新日志 → 向用户确认（需要更新日志吗？）
  → 用户确认 → 汇报总结
```

---

### 阶段一：代码审查

1. **需求符合度**：读取用户在上下文中发送的项目要求，逐项检查代码是否符合要求，功能无缺漏、内容无丢失。
2. **代码质量**：检查项目代码是否有漏洞/bug，包括但不限于安全方面、功能方面、样式方面。
3. **UI/UX 还原度**：若涉及界面改动，务必对照上下文中的设计稿描述或相关截图，检查布局、间距、颜色及交互逻辑的像素级还原度。

详细检查清单见 `references/review-checklist.md`。

审查完毕后，按严重程度汇总发现的问题，然后**立即向用户确认**：
- 议题："代码审查发现 N 个问题（X 严重 / Y 警告 / Z 建议），是否需要修复？"
- 选项：全部修复 / 仅修复严重+警告 / 暂不处理

**在用户明确回复之前，不得进入阶段二。**

---

### 阶段二：版本号决策

**仅在用户对阶段一确认后，才进入此阶段。**

根据变更性质按 `references/version-decision.md` 的规则计算出建议版本号，然后**立即向用户确认**：
- 议题："根据本次变更（XXX），建议版本号从 vA.B.C.D 变更为 vX.Y.Z.B，是否同意？"
- 选项：同意，按此版本号 / 不同意，手动指定

**在用户明确回复之前，不得进入阶段三。**

---

### 阶段三：前端同步

**仅在用户对阶段二确认后，才进入此阶段。**

检查项目中是否存在需要同步版本号的文件（如 `package.json`、`AndroidManifest.xml`、`build.gradle` 等）。若存在，**立即向用户确认**：
- 议题："检测到以下文件需要同步版本号：[列出文件]，是否同步修改？"
- 选项：同步所有文件 / 跳过，暂不修改

若无需要同步的文件，直接跳过此阶段，进入阶段四。

**在用户明确回复之前，不得进入阶段四。**

---

### 阶段四：更新日志

**仅在用户对阶段三确认后（或阶段三跳过），才进入此阶段。**

检查项目中是否存在更新公告（如首次启动弹窗）或更新历史（如关于页更新日志）。若存在，**立即向用户确认**：
- 议题："检测到项目有更新日志/公告，是否需要更新以反映本次变更？"
- 选项：更新，内容如下：XXX / 跳过，暂不更新

若无更新日志或公告，直接跳过此阶段，进入汇报总结。

---

### 最终：汇报总结

所有阶段确认完毕后，按 `references/reporting-standards.md` 汇总输出最终结果。

## 子场景路由

若用户意图更聚焦，可路由到专项场景：

- 用户只关心代码逻辑 → `scenes/code-review.md`
- 用户只关心 UI 还原 → `scenes/ui-review.md`
- 用户只关心安全问题 → `scenes/security-audit.md`
